# Evaluation

本项目的评估重点不是单一数据集分数，而是系统是否能稳定完成多模态证据问答任务。公开 benchmark 可以作为参考数据源，但项目本身应围绕系统能力进行评估。

## What We Evaluate

1. 检索覆盖率  
   正确证据是否进入候选池。

2. 重排准确性  
   最相关的文本、图片、表格或图注是否排在前列。

3. 图文一致性  
   `<PIC>` 或前端图片证据是否真正支撑回答。

4. 答案忠实度  
   答案是否基于证据，是否避免编造。

5. 证据链质量  
   证据之间是否能形成清晰的推理路径。

6. 端到端体验  
   上传 PDF、提问、等待任务、查看答案和证据是否顺畅。

## Useful Local Checks

本地脚本可以检查结构性问题：

```bash
python scripts/05_evaluate.py
python scripts/12_evaluate_evidence_chains.py
python scripts/14_chunk_quality_report.py
```

这些脚本适合发现：

- 检索结果中是否覆盖正确证据。
- 重排后的证据是否更靠前。
- 证据链是否包含文本、表格、图片或图注等关键模态。
- chunk 是否过长、过短或结构信息不足。

但它们不能替代真实质量评估，因为离线指标无法完全判断答案是否真的好、图片是否真的帮助理解。

## Benchmark Data

本地 benchmark 数据可放在：

```text
data/sample/

```

这些数据用于调试和验证系统能力。正式项目说明中应把外部数据集视为参考数据，而不是项目目标本身。

## Known Evaluation Risks

- 图片数量多不等于图文互补好。
- query expansion 可能提高召回，也可能引入无关证据。
- LLM 自检可能修正答案，也可能把具体证据改成泛化话术。
- 本地启发式规则和外部 LLM 评分不一定一致。
- 全量自动优化风险高，推荐小范围、可回滚的实验。

## Recommended Practice

- 保留一个稳定基线。
- 每次只改变一个主要变量。
- 记录输入、参数、输出文件和观察结果。
- 对低分或异常结果做样例级对比，而不是只看平均分。
- 对图片证据单独抽查，确认图片与问题动作、部件或表格直接相关。
