# 面向复杂文档问答的查询感知跨模态节点重要性重排序方法研究执行清单

本清单完全按《开题报告.docx》的范围整理：基于 RAG-Anything 或简化解析流程，围绕复杂 PDF 文档问答做候选证据节点重排序。重点不是重新做一个完整多模态 RAG 框架，也不是训练大模型，而是把“初步检索出来的证据”重新排序，让关键文本、表格、图表和页面证据更靠前。

## 0. 最终要做成什么

一句话目标：

用户上传一份图文表混排 PDF，系统先找出候选证据，再用 Sim、查询感知 PPR、Bridge 三种分数重新排序，最后展示重排序前后 Top-K 证据变化、答案、证据页码和节点关系。

最终交付物：

- 2-3 份复杂 PDF 文档。
- 20-30 个测试问题，带标准答案和正确证据标注。
- 一个最小可行的多模态文档问答流程。
- 一个重排序模块：语义相似度 + 查询感知 PPR + Bridge 分数。
- 四组实验结果：G0 原始排序、G1 语义排序、G2 语义 + PPR、G3 语义 + PPR + Bridge。
- 指标表和对比图：Recall@K、MRR、nDCG@5、Evidence Hit Rate、Modality Hit Rate、Citation Accuracy、Rerank Time。
- Demo：上传 PDF、输入问题、展示 Top-K 变化、答案、证据页码和关系。
- 结题报告、PPT、答辩讲稿、README、代码说明。

最低成功标准：

- 至少 2 份 PDF。
- 至少 20 个问题。
- 至少跑通 G0-G3 四组排序结果。
- 至少展示 3 个成功案例和 2 个失败案例。
- Demo 可以稳定演示一轮完整流程。

### 第 2 周第 2 天：实现查询感知 PPR，约 5-8 小时

陈耀洋：

- 把节点和边读成图。
- 对当前问题，从初步 Top-K 候选里选种子节点。
- 种子节点就是“这个问题一开始检索到的节点”。
- 用 Personalized PageRank 计算每个候选节点的重要性。

通俗理解：

- 普通 PageRank 问的是“整张图里谁重要”。
- 查询感知 PPR 问的是“围绕这个问题，哪些相关节点更重要”。
- 如果一个节点和问题相关，而且在图结构上连接到很多相关证据，它的 PPR 分数就应该更高。

G2 公式：

```text
Score_1(v | q) = alpha * Sim(q, v) + beta * PPR_q(v)
```

建议第一版参数：

```text
alpha = 0.7
beta = 0.3
```

如果 PPR 分数太小，先把 Sim 和 PPR 都归一化到 0-1 再融合。

赵依然：

- 找 3 个问题人工看 G2 排序有没有变合理。
- 尤其看正确证据有没有往前移。

刘莉：

- 建一列 `rank_change_g1_to_g2`。

柯熙钰：

完成标准：

- 至少有 1-2 个案例能看到正确证据前移。

### 第 2 周第 3 天：实现 Bridge 分数，约 5-8 小时

陈耀洋：

- 对每个候选节点计算 Bridge 分数。
- Bridge 关注的是：这个节点是不是连接了多种模态的相关证据。

按开题报告实现：

```text
Bridge*(v | q) = TypeWeight(v) * DegreeNorm(v) * CrossModalRel(v | q)
DegreeNorm(v) = 1 / log(2 + deg(v))
```

第一版 TypeWeight：

```text
text = 1.0
table = 1.2
figure = 1.2
caption = 1.0
page = 0.5
title = 0.6
```

这样做的原因：

- 表格和图表往往是关键证据，稍微加权。
- 页面和标题是结构节点，不能让它们因为连接多就一直排第一。

CrossModalRel 简化做法：

- 看候选节点的邻居。
- 如果邻居和问题相似度超过阈值 `tau`，就认为这个邻居和问题相关。
- 统计相关邻居覆盖了几种模态。
- 覆盖模态越多，Bridge 越高。

建议第一版：

```text
tau = 0.2
M = 4
模态类型 = text, table, figure, page
```

赵依然：

- 对表格题和图表题重点检查 Bridge 是否把表格、图表、图注推上来。

刘莉：

- 增加 `bridge_score` 列。
- 做一张案例表：同一问题下，每个节点的 Sim、PPR、Bridge 分数分别是多少。

柯熙钰：

- 做 Bridge 的答辩解释图：正文、表格、图注、页面之间的桥接关系。

完成标准：

- 每个候选节点都有 Bridge 分数。
- Bridge 不应该让 page 节点长期霸榜。如果出现这种情况，降低 page 的 TypeWeight。

### 第 2 周第 4 天：实现 G3 融合排序，约 4-6 小时

陈耀洋：

- 实现最终融合公式：

```text
Score(v | q) = lambda_s * Sim(q, v) + lambda_p * PPR_q(v) + lambda_b * Bridge*(v | q)
```

建议第一版参数：

```text
lambda_s = 0.6
lambda_p = 0.25
lambda_b = 0.15
```

为什么这样设：

- 初期数据少，语义相似度最稳定，所以权重大一点。
- PPR 用来补结构关系。

## 8. 四组实验具体怎么做

### G0：RAG-Anything 原始排序

含义：

- 不做二次排序。
- 初步检索给什么顺序，就保留什么顺序。

怎么做：

- 保存初步检索的 Top-K。
- 作为对照组。

输出：

```text
question_id, method=G0, rank, node_id, score
```

### G1：仅语义相似度排序

含义：

- 只看问题和节点内容像不像。

怎么做：

- 计算 `Sim(q, v)`。
- 从高到低排序。

适合：

- 文本事实题通常比较有效。

不足：

- 表格、图表、图注和正文之间需要结构关系时，可能不够。

### G2：语义相似度 + 查询感知 PPR

含义：

- 不只看像不像，还看这个节点在当前问题相关的局部图里重不重要。

怎么做：

- 用 G0 或 G1 的 Top-K 作为种子。
- 在节点图里跑 PPR。
- 融合 Sim 和 PPR。

输出：

```text
score_g2 = alpha * sim + beta * ppr
```

### G3：语义相似度 + PPR + Bridge

含义：

- 在 G2 基础上，再奖励跨模态桥接节点。

怎么做：

- 计算 Bridge。
- 融合 Sim、PPR、Bridge。

输出：

```text
score_g3 = lambda_s * sim + lambda_p * ppr + lambda_b * bridge
```

这是本课题最终方法。

## 10. 指标怎么算，通俗版

Recall@K：

- 正确证据有没有出现在前 K 个里。
- 如果正确证据在 Top-5，Recall@5 就算命中。

MRR：

- 正确证据越靠前越好。
- 第 1 名得 1 分，第 2 名得 0.5 分，第 3 名得 0.333 分。

nDCG@5：

- 衡量 Top-5 的整体排序质量。
- 正确证据靠前，分高。
- 多个正确证据都靠前，分更高。

Evidence Hit Rate：

- 是否命中正确证据。

Modality Hit Rate：

- 是否命中正确模态。
- 比如表格题应该命中 table，图表题应该命中 figure 或 caption。

Citation Accuracy：

- 页码、图号、表号是否引用正确。

Rerank Time：

- 重排序多花了多少时间。
- 用来证明方法轻量。

## 12. 风险和简化方案

风险 1：RAG-Anything 完整部署太难。

处理：

- 最多花 1 天尝试。
- 跑不通就走简化解析。
- 开题报告已经允许“RAG-Anything 或简化解析方式获得初步检索结果和候选证据节点”。

风险 2：图表内容无法自动理解。

处理：

- 第一版把图表节点内容设为图注加人工摘要。
- 图表题主要验证“图表节点和相关正文、图注的排序”。

风险 3：正确证据没有进入初步 Top-K。

处理：

- 这是召回问题，不是重排序问题。
- 记录为失败案例。
- 说明重排序只能调整候选池内部顺序，不能拯救完全没召回的证据。

风险 4：Bridge 把页面节点排太高。

处理：

- 降低 page 的 TypeWeight。
- 对高度连接节点使用 DegreeNorm。
- 在案例分析中说明为什么要做类型约束和度数归一化。

风险 5：指标不提升。

处理：

- 分题型看，不只看总平均。
- Bridge 可能主要提升表格题、图表题和图文一致性题。
- 如果总分不明显，也可以用案例证明方法在跨模态问题上有效。
